# Architectural refactoring plan

- [x] ARCH-001: [P1] Type resolution is carrying too many architectural roles. typeResolver.ts (line 33) handles cache policy, cycle detection, warning buffering, external-type policy, TS internals, and every AnyType construction path. The long ordered branch chain around type flags (line 168) makes new type support risky because behavior depends on branch order. A TypeResolutionPipeline pattern would help: small resolvers like UnionResolver, ObjectResolver, ExternalResolver, ConditionalResolver, coordinated by one session object that owns cache, cycle guards, and diagnostics.
- [x] ARCH-002: [P1] ParserContext is a mutable ambient state bag. It exposes stacks and substitutions directly in parser.ts (line 96), then many parsers manually push/pop nodes and symbols. Warning locations later depend on that stack in typeResolver.ts (line 626). Introduce scoped parser-context helpers for diagnostic symbols, diagnostic source nodes, and type-parameter substitutions; that would make stack correctness local and remove a lot of fragile finally boilerplate.
- [x] ARCH-003: [P1] Export parsing has become a second orchestration hub. exportParser.ts (line 19) does export-symbol targeting, namespace merging, re-export metadata, type acquisition, declaration dispatch, extends extraction, documentation, warning emission, and model construction. A stronger pattern would be: first normalize exports into an ExportDescriptor, then convert descriptors into ExportNodes. That would also remove repeated “append namespace members” paths and make alias/re-export behavior testable in isolation.
- [x] ARCH-004: [P2] Function and class signature parsing are duplicated and already diverge. functionParser.ts (line 69) and classParser.ts (line 257) both parse parameters, docs, visibility, optionality, defaults, and return types. They differ on non-literal default values: class parameters keep initializer.getText() while function parameters do not. Expand signatureParser.ts into the single owner of parseCallSignature, parseParameter, parseReturnType, and parameter docs.
- [x] ARCH-005: [P2] Model objects mix DTOs, normalization, equivalence, and rendering. UnionNode normalizes in its constructor, while compoundTypeUtils.ts (line 188) owns a large structural-equivalence algorithm that imports nearly every model. This creates circular pressure and makes every new type require updates across models, resolver, and dedupe logic. Consider immutable model DTOs plus a TypeFactory/TypeCanonicalizer and a separate TypeEquivalence visitor.
- [ ] ARCH-006: [P2] TypeScript private/internal API usage is scattered. Examples include checker.getElementTypeOfArrayType in typeResolver.ts (line 210), type.origin in union resolution, mapper probing in objectParser.ts (line 145), and node_modules string checks in typeResolver.ts (line 538). A CompilerFacade would isolate these version-sensitive calls and make TypeScript upgrades much safer.
- [x] ARCH-007: [P2] React component detection is a post-parse heuristic transform. componentParser.ts (line 22) rewrites FunctionNode exports to ComponentNode based on capitalization and return-type regexes. That works, but it is domain policy embedded after generic parsing. A transform/classifier pipeline would make this explicit and configurable, and ExportNode.withType(...) would avoid losing future metadata when rewriting nodes.
- [x] ARCH-008: [P3] Diagnostics and documentation are inconsistent. Most warnings go through onWarning, but default-export symbol failure logs directly with console.error in exportParser.ts (line 207). Parameter JSDoc parsing is duplicated outside documentationParser. Also, Documentation.getTagValue returns the tag name instead of the value in documentation.ts (line 13), which looks like a small real bug.
- [x] ARCH-009: [P3] Tests are broad but pattern-level tests are thin. The fixture suite is strong: 111 golden input/output cases, wired in test/index.test.ts (line 10). As you extract the patterns above, add small unit tests for ExportDescriptor, signature parsing, diagnostics scope, canonicalization, and compiler-adapter fallbacks. Also move the duplicated in-memory program helper from test/index.test.ts (line 72) and test/warnings.test.ts (line 44) into test support.
